<objective>
Fix TUI input bar disabled-state race condition and make WS auth failures visible to the user.

Purpose: The WebSocket client fires `_on_state_change("connecting")` before `ChatScreen` is mounted, causing `query_one` to throw `NoMatches` which is silently swallowed. This leaves InputBar enabled during connection (wrong) and hides auth failures from the user entirely.

Output: Reliable InputBar enable/disable tied to WS state, visible error messages on auth failure and reconnection.
</objective>

**app.py** -- Replace `_on_state_change` (lines 232-238) with a two-method pattern:

1. `_on_state_change(self, state: str)` becomes a thin dispatcher that defers DOM work:
```python
def _on_state_change(self, state: str) -> None:
    self.call_after_refresh(self._apply_state_to_ui, state)
```

2. New `_apply_state_to_ui(self, state: str)` does the actual DOM mutations with per-widget try/except (not one blanket catch):
```python
def _apply_state_to_ui(self, state: str) -> None:
    try:
        self.screen.query_one("#status-bar", StatusBar).set_status(
            session=self._session_key, state=state,
        )
    except Exception:
        pass
    try:
        self.screen.query_one("#input-bar", InputBar).disabled = state != "connected"
    except Exception:
        pass
    # Surface auth failure and reconnection in chat log
    if state == "auth_failed":
        try:
            self.screen.query_one("#chat-log", ChatLog).append_error(
                "Gateway authentication failed. Check that the gateway is "
                "running and ~/.yigthinker/gateway.token is valid."
            )
        except Exception:
            pass
    elif state == "reconnecting":
        try:
            self.screen.query_one("#chat-log", ChatLog).append_error(
                "Gateway connection lost. Reconnecting..."
            )
        except Exception:
            pass
```

Also remove the redundant `_status_bar.set_status` call from `on_mount` (lines 71-73) since the `connect_loop` worker fires `_set_state("connecting")` immediately, which will now reliably reach the UI via `call_after_refresh`.
